# 业务级 Agent 上下文

本文档用于帮助 agent 从业务视角理解普悦业财一体 CRM 销管系统。它不是接口文档，也不是需求冻结文件；当代码与本文冲突时，以当前主应用代码和用户最新要求为准。

## 业务闭环

系统当前围绕以下闭环建设：

```text
客户/渠道 -> 线索 -> 商机 -> 项目 -> 合同 -> 工单/派工/装机 -> 评价/复盘 -> 报表/目标调整
```

销售目标管理贯穿闭环，用年度、季度、月度任务约束销售动作，并通过实际业绩和报表反馈完成情况。

## 核心业务对象

| 对象 | 业务含义 | 典型动作 |
|------|----------|----------|
| 客户 | 终端客户或采购主体 | 建档、维护、查看全景、关联渠道、查看财务视图 |
| 渠道 | 合作伙伴、代理或来源渠道 | 建档、联系人维护、分配、培训、绩效统计 |
| 线索 | 初始销售机会来源 | 新增、跟进、筛选、转商机 |
| 商机 | 已确认的销售机会 | 阶段推进、金额维护、转项目 |
| 项目 | 商机落地后的项目过程 | 维护进度、关联合同、推进交付 |
| 合同 | 商务成交与回款依据 | 合同管理、回款进度统计 |
| 销售任务 | 年度/季度/月度目标 | 制定、拆解、更新、删除、录入实际业绩、查看完成率 |
| 工单 | 售后、实施、安装或现场服务任务 | 创建、派工、状态同步、评价 |
| 产品 | 销售和装机对象 | 产品维护、实体产品、装机记录 |
| 跟进记录 | 销售或渠道过程记录 | 记录拜访、沟通、下一步计划 |
| 日报/周报 | 围绕日常动作沉淀的结构化工作报告 | 生成草稿、补充备注、提交、撤回、团队查看 |
| 通知 | 面向用户的站内消息 | 查看未读、标记已读、跳转关联业务对象 |
| 待办 | 从业务对象派生的待处理事项 | 聚合跟进、合同、工单、日报/周报、离职交接并跳转处理 |
| 离职交接 | 离职或停用用户的业务资产转移流程 | 预览资产、指派接收人、执行交接、取消请求 |

## 角色视角

| 角色 | 关注点 |
|------|--------|
| 管理员 | 用户、权限、字典、告警、操作日志、全量数据 |
| 业务管理者 | 销售目标、团队业绩、销售漏斗、项目合同进展 |
| 销售人员 | 自己负责的客户、线索、商机、项目、合同、销售任务 |
| 财务人员 | 客户财务视图、合同、回款进度、财务相关数据 |
| 技术员 | 分配给自己的工单、派工状态、装机和服务评价 |
| 渠道运营 | 渠道档案、渠道跟进、渠道绩效、培训和客户关联 |

## 销售任务管理口径

销售任务管理是当前重点能力，开发或验收时应覆盖：

- 年度任务：创建、更新、删除、目标金额、负责人、期间归属。
- 季度任务：从年度目标拆解或独立维护，校验归属年度和金额规则。
- 月度任务：从季度目标拆解或独立维护，校验归属季度和金额规则。
- 实际业绩：支持录入与统计，能够计算完成率和进度。
- 列表/树视图：支持按人员、年份、周期层级查看。
- 规则校验：避免非法时间层级、越界金额、重复或不一致数据。
- 权限：销售人员只能操作授权范围内数据，管理角色可管理全局。
- 测试：后端规则测试、流程测试、前端类型和构建验证都应通过。

## 下一阶段业务目标

下一阶段重点建设尚未落地的模块，并继续完善已上线的日报/周报、角色化工作台、通知中心、待办中心和离职交接：

- 报价/价格/方案管理：补齐商机到合同之间的报价过程，支持报价明细、版本、折扣规则、报价转合同。
- 附件与文档管理：为客户、商机、项目、合同、报价、工单等对象提供统一附件能力。
- 数据导入导出：支持模板下载、客户/线索/销售目标导入、错误报告和导入任务追踪。
- 客户联系人与组织关系深化：支持客户多联系人、主联系人、决策链、集团/分子公司关系。
- 通知与待办增强：在现有站内通知和派生待办基础上，补齐定时提醒、飞书外发、订阅规则、日历视图、完成/延期/关闭动作。
- 日报/周报增强：优化提醒策略、统计口径、管理入口和提交规则。当前已支持销售、渠道运营、技术服务等角色，管理者可查看团队日报/周报；暂不强制提交，财务第一版不参与日报/周报。
- 角色化工作台增强：当前已启用角色工作台入口，后续继续补齐管理员、业务管理者、销售、财务、技术员、渠道运营的差异化首页内容。
- 离职交接增强：在现有管理员入口基础上补齐部门负责人/当事人视图、审批协同与运维说明。
- 渠道运营角色：新增独立角色，建议编码为 `channel_ops`。
- 部门负责人关系：新增通用字段 `department_manager_id`，用于日报/周报团队视图和角色化工作台团队范围；`sales_leader_id` 仅作为历史销售字段兼容。

## 角色化工作台目标

| 角色 | 工作台重点 |
|------|------------|
| 管理员 | 系统健康、用户、权限、告警、操作日志 |
| 业务管理者 | 销售目标完成、团队业绩、销售漏斗、重点商机、逾期事项 |
| 销售人员 | 我的客户、我的线索/商机、销售任务、待跟进、日报周报 |
| 财务人员 | 合同、回款进度、逾期提醒、客户财务视图；第一版不参与日报/周报 |
| 技术员 | 我的工单、派工状态、SLA、装机和服务评价 |
| 渠道运营 | 独立角色 `channel_ops`，展示渠道绩效、渠道跟进、渠道培训、渠道关联客户、日报周报 |

当前工作台与日报/周报启用边界：

- 后端入口：`backend/app/routers/dashboard.py`、`backend/app/routers/work_report.py`
- 前端入口：`frontend/src/pages/MyDashboard.tsx`、`frontend/src/pages/WorkReportPage.tsx`
- 团队范围优先使用 `department_manager_id`
- 第一版默认不把 `finance` 纳入日报/周报提交流程

当前通知、待办与离职交接启用边界：

- 通知中心入口：`backend/app/routers/notification.py`、`frontend/src/pages/NotificationCenterPage.tsx`
- 待办中心入口：`backend/app/routers/todo.py`、`frontend/src/pages/TodoCenterPage.tsx`
- 离职交接入口：`backend/app/routers/handover.py`、`frontend/src/pages/HandoverListPage.tsx`、`frontend/src/pages/HandoverDetailPage.tsx`
- 待办中心首版为派生待办，不使用通用 `todos` 表。
- 离职交接首版由 `admin` 执行分配、执行和取消；部门负责人/当事人视图后续增强。

## 当前启用边界

启用能力以主应用入口为准：

- 后端：`backend/app/main.py`
- 前端：`frontend/src/App.tsx`

以下模块代码存在但不应默认承诺为已启用能力：

- `backend/app/routers/financials.py`
- `backend/app/routers/kingdee_integration.py`
- 其他未在 `backend/app/main.py` 注册的历史 Router

如果用户要求启用这些能力，需要补齐：

- Router 注册
- 权限策略
- Schema 和输入校验
- 前端入口或调用链
- 测试覆盖
- 部署与环境变量说明

## 业务开发检查清单

每次业务功能开发结束前，至少确认：

- 后端接口是否有认证、权限、输入校验和错误处理。
- 列表接口是否有分页上限。
- 前端是否有加载、空状态、错误提示和权限控制。
- 数据模型、Schema、API 类型是否一致。
- 是否补充了针对核心规则的测试。
- 团队范围是否使用 `department_manager_id`，而不是继续绑定销售专用 `sales_leader_id`。
- README、计划文档或完成报告是否需要同步更新。
- 未提交真实密钥、备份数据、运行时产物或无关大文件。

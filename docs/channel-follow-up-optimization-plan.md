# 渠道管理跟进记录优化方案

## 背景与问题

当前 CRM 系统中的跟进记录（FollowUp）功能存在业务场景混淆问题：

### 问题描述
- **业务跟进记录**: 用于线索、商机、项目的销售流程跟踪，需要填写"跟进结论"（如：已签约、商务谈判中、已流失等）
- **渠道跟进记录**: 主要用于渠道拜访记录（拜访记录），"跟进结论"字段不适用

### 当前实现问题
1. **单一模型约束**: `follow_ups` 表中的 `follow_up_conclusion` 字段为必填项
2. **字段不匹配**: 渠道拜访场景被强制要求填写销售导向的结论字段
3. **用户体验割裂**: 渠道管理人员需要填写与其工作无关的业务字段

## 解决方案

采用**单一表扩展 + 动态字段显示**方案，在保持现有数据结构的基础上进行优化。

### 核心原则
- ✅ **不新建表** - 避免数据迁移和API重构
- ✅ **字段扩充** - 添加类型区分和场景专用字段  
- ✅ **动态显示** - 前端根据场景智能显示相关字段
- ✅ **向后兼容** - 保持现有API接口不变

## 技术实施方案

### 1. 数据库变更

#### 新增字段
| 字段名 | 类型 | 说明 | 约束 |
|--------|------|------|------|
| `follow_up_type` | VARCHAR(20) | 跟进类型标识 | NOT NULL, 默认 'business' |
| `visit_location` | VARCHAR(100) | 拜访地点 | NULLABLE |
| `visit_attendees` | VARCHAR(255) | 参与人员 | NULLABLE |
| `visit_purpose` | VARCHAR(100) | 拜访目的 | NULLABLE |

#### 字段约束调整
| 原字段 | 调整后 | 说明 |
|--------|--------|------|
| `follow_up_conclusion` | NULLABLE | 不再强制必填，根据类型条件验证 |

### 2. 后端逻辑

#### 类型推断规则
```python
# 自动推断 follow_up_type
if channel_id is not None and all_business_ids is None:
    follow_up_type = 'channel'
elif any_business_id is not None and channel_id is None:
    follow_up_type = 'business'
else:
    # 混合场景，按创建时指定的类型处理
    follow_up_type = specified_type
```

#### 条件验证逻辑
```python
if follow_up_type == 'business':
    # 业务跟进：结论字段必填
    if not follow_up_conclusion:
        raise HTTPException(400, "业务跟进必须填写跟进结论")
elif follow_up_type == 'channel':
    # 渠道跟进：结论字段可选，可使用新字段
    pass
```

#### API 保持兼容
- 所有现有 API 端点保持不变
- 新字段为可选参数，不影响现有调用
- 自动类型推断确保无缝过渡

### 3. 前端优化

#### FollowUpModal.tsx 条件渲染
| 场景 | 显示字段 | 隐藏字段 |
|------|----------|----------|
| **业务跟进**<br>(关联线索/商机/项目) | - 跟进结论<br>- 下一步行动<br>- 下次跟进日期<br>- 跟进内容 | - 拜访地点<br>- 参与人员<br>- 拜访目的 |
| **渠道跟进**<br>(关联渠道ID) | - 拜访地点<br>- 参与人员<br>- 拜访目的<br>- 跟进内容 | - 跟进结论<br>- 下一步行动<br>- 下次跟进日期 |

#### 表单标题优化
- 业务跟进: "新建跟进记录"
- 渠道跟进: "新建渠道拜访记录"

### 4. 字典配置

#### 新增字典项
- **拜访目的**: 商务洽谈、技术支持、关系维护、产品培训、市场调研等
- **拜访方式**: 现场拜访、电话沟通、视频会议等

#### 保留现有字典
- **跟进结论**: 保持现有的销售流程结论选项
